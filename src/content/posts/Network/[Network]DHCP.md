# DHCP(Dynamic Host Configuration Protocol)
앞서 OSI7 Layer를 생각해봤을때, 내가 PC를 한대 구매했다. 그리고 네트워크 연결을 해야한다라고
가정했을때, 우리는 한번도 저렇게 복잡한 과정의 동작원리를 생각한적이 없다.  

알아서 PC에 랜선만 갖다가 꽂으면 알아서 IP주소가 할당되고, 서브넷 마스크도 생기고, 내 IP대역의 게이트웨이도 만들어진다.  
즉, `자동으로 알아서 할당`된다는 것이다.  

이를 통해서 네트워크 관지자, 엔지니어들이 일일이 IP주소를 설정할 필요 없이, 네트워크에 접속한 장치들이 자동으로
적절한 네트워크를 설정 받는다는 것이다.  이를 `중앙집중식으로 DHCP서버`에서 관리를 하게된다.

## 동작원리

DHCP를 쉽게 이해한 바로는 `알잘딱`하게 네트워크 환경을 구성해준다는 것인데 4가지 단계로 이를 정의할 수 있다.

1. Discover(탐색)
   - 새롭게 연결된 클라이언트가 DHCP 서버가 있는지 확인하기 위해서 Broadcast 메시지(예: 나 IP 줘)를 던진다.
2. Offer(제공)
    - DHCP서버는 클라이언트의 요청을 받고, 사용 가능 한 IP와 기타 네트쿼으 설정을 포함하여 제안 메시지를 새롭게 연결된 클라이언트에게 보낸다.
3. Request(요청)
   - 새로운 클라이언트는 제안중 하나의 IP주소를 선택하고, 그 IP 주소를 사용하겠다는 메시지를 요청메세지로 보낸다.
4. Acknowledgment(승인)
   - 서버는 요청을 승인하게되고 선택한 IP 주소와 네트워크 설정을 클라이언트에게 할당한다.

> 💡 Broadcast 요청
> 
> 광역 도발이다. 내가 가진 네트워크 대역에 광역으로 신호를 날리는 것이다.
> 그게 나의 네트워크 대역의 가장 마지막 IP주소를 의미한다.

어디서 많이 본 구조가 아닌가 싶다.

TCP의 3way handshake와 비슷해보인다. 하지만, 처음에 DHCP서버를 Broadcast로 찾아야하는 것이 다르다.
> Client: 나 여기 있는 친구들이랑 놀래. 방장 누구야? 알려줘! 
> DHCP Server: 오 마침 잘 왔어. 너 나랑 연결할꺼면 이 꽃문양 종이컵으로 해!
> Client: 좋아. 잘 들려?
> DHCP Server: 응 잘 들려!

이 방식인것 같다. 사실 네트워크를 연결하는 방법은 복잡하다. IP도 할당해주고, 서브넷도 할당해주고
중앙집중식 DHCP서버를 활용한다면, `알잘딱`하게 IP와 서브넷 마스크를 셋팅 해줄것이다.

그렇기때문에 대부분 기업용에서 많이 활용하게 된다. 네트워크 관리도 용이하고 중복된 IP를 방지할 수 있다.
그리고 누가 어떤 IP를 쓰고 있는지도 확인 할 수 있을 것이다.
